## **OAuth의 핵심 참여자와 상호 작용**

| 항목          | 용어                                      | 설명                                                       |
| ------------- | ----------------------------------------- | ---------------------------------------------------------- |
| 나의 서비스   | **Client**                                | 사용자를 대신하여 다른 서비스의 데이터를 이용하려는 서비스 |
| 사용자        | **Resource Owner**                        | 서비스의 주인이자 권한을 부여하는 주체                     |
| 그들의 서비스 | **Resource Owner & Authorization Server** | 실제 데이터를 가지고 있는 대형 서비스                      |

> OAuth는 나의 서비스가 사용자를 대신하여 그들의 서비스에 접근해 데이터를 읽거나 쓰기 위해 사용된다.

**IN SHIELD ID..**

- `클라이언트(Client)` : 사용자가 로그인하려는 애플리케이션
- `리소스 사용자(Resource Owner)` : 웹/앱을 사용하는 사용자
- `인증서버(Authorization Server)` : 권한 발급 서버 (토큰 발행 및 갱신 서버)
- `리소스 서버(Resource Server)` : 가져오고자하는 정보가 있는 서버 (API)

## **기존 방식의 한계와 보안적 필요성**

과거에는 다른 서비스의 기능을 이용하기 위해 사용자로부터 직접 아이디와 비밀번호를 전달받아 저장하는 방식을 사용했었으나, 이는 아래와 같은 보안 문제들을 야기하는 점이 존재한다.

- **사용자 입장**: 사용자 입장: 처음 보는 서비스에 자신의 소중한 계정 정보를 믿고 맡기기 어렵다. 또한 대다수 사용자가 여러 서비스에서 동일한 비밀번호를 사용하므로, 정보 유출 시 연쇄적인 보안 사고로 이어질 위험이 크다.
- **나의 서비스 입장**: 타인의 민감한 계정 정보를 보관하고 관리해야 하는 막대한 책임과 유출 시의 위험을 부담해야 한다.
- **그들의 서비스 입장**: 신뢰할 수 없는 제3자가 자사 사용자의 계정 정보를 직접 소유하는 상황은 매우 불만족스럽고 위험한 상태

> **OAuth 2.0은 이러한 보안적 위협으로부터 모두를 보호하기 위해 등장한 표준 기술**

## **액세스 토큰(Access Token): OAuth 2.0의 심장**

OAuth 2.0의 가장 큰 특징이자 목적은 아이디와 비밀번호 대신 **액세스 토큰(Access Token)**을 발급받아 사용하는 것. 이 토큰은 두 가지 결정적인 장점을 가진다.

- **비밀번호가 아님**: 사용자의 실제 계정 비밀번호가 아니므로 유출 시의 위험이 상대적으로 적다.
- **제한적 권한 부여**: 해당 서비스의 모든 기능이 아니라, 나의 서비스가 작동하는 데 꼭 필요한 필수 기능만 부분적으로 허용하는 비밀번호 역할을 한다.

> 따라서 OAuth 2.0 시스템의 핵심 원리는 사용자의 요청에 따라 '그들의 서비스'가 '나의 서비스'에 이 액세스 토큰을 안전하게 전달하는 과정을 설계하는 것

## **확장된 활용: 연합 아이덴티티(Federated Identity)**

OAuth 2.0은 단순히 데이터를 가져오는 것을 넘어, 회원의 아이디와 비밀번호를 직접 보관하지 않고도 사용자를 식별할 수 있는 로그인 기능으로 확장된다.

-> 우리가 흔히 접하는 '구글로 로그인', '페이스북으로 로그인'과 같은 기능이 바로 이 **연합 아이덴티티(Federated Identity)** 기술에 해당하며, 그 기반에는 OAuth 2.0이 자리 잡고 있다.
